## テンプレートルール

### 基本構造
- `YYYY-MM-DD` は実行日の日付に置換する
- `N件` は各カテゴリの実際の採用記事数に置換する（目標件数ではなく実数）
- カテゴリの順序は固定：ビジネス → AI → 新サービス → 国内技術 → EM/PM → セキュリティ
- この順序と見出し絵文字は勝手に変更しない

### 件数と「本日の更新なし」の扱い
- `SOURCES.md` に記載された件数レンジは**上限の目安**であり、下限の達成義務ではない
- 下限を満たせない場合でも、無理に埋めず実数で出す（件数優先より品質優先）
- カテゴリ内に該当記事が **0件** の場合は、以下のように扱う（EM/PM に限らず全カテゴリ共通）
  - `## 🎯 今日のまとめ` 内の該当カテゴリ見出し下に「本日の更新なし」と記載する
  - そのカテゴリの `<details>` ブロックは**丸ごと省略する**（空の details を残さない）

### 要約の書き方
- 各カテゴリのまとめ本文は **1〜3文** の事実ベースの短い段落にする
- まとめ本文で言及する記事は、必ず同カテゴリの `<details>` 側にも登場していること（片方だけに出てくる記事を作らない）
- `<details>` 内の各記事は「**見出し**」＋ **1〜2文の要約** ＋ 元記事リンク の構成
- 要約は事実ベース。推測・感想・評価語（「注目」「話題」など）は極力避ける
- `last30days-skill` 由来の反応補助ソースを使う場合は、記事要約とは分けて「補足:」「コミュニティ反応:」として書く
- 反応補助ソースだけで、通常ニュース本文の事実・数字・発表内容を断定しない

### 英語ソース記事の詳細要約（📖 詳しく読む）
- 英語ソースの記事には、1〜2文の通常要約に加えて `<details><summary>📖 詳しく読む</summary>...</details>` の折り畳みブロックを付ける
- **対象ソース（`SOURCES.md` で言語=EN のもの）**: Reuters, TechCrunch, OpenAI Blog, Google Research Blog, Anthropic Blog, Hacker News, Product Hunt, LeadDev, The Hacker News
- **文字数上限（厳守）**: 1記事あたり **200字以内・最大3文**。これを超えるとトークン消費が膨張するため、必ず収める
- **残す情報**: 数字・固有名詞・結論の3点のみ
- **残さない情報**: 前置き、背景の説明、技術的な仕組みの詳述、推測、感想、評価語（「注目」「重要」「画期的」など）
- **書き方ルール**:
  - 事実ベースを維持する（推測・意見を入れない）
  - 全文転載・逐語訳はしない。要点の極小再構成として書く
  - 日本語ソース（nikkei.com, newspicks.com, dev.classmethod.jp, forest.watch.impress.co.jp, ipa.go.jp）の記事には付けない
- **構造**: 記事リンクの直後に `<details>` をネストする。カテゴリ単位の `<details>` の中にさらに記事単位の `<details>` が入る形になる

#### 200字以内の良い例
> CVE-2026-48172はLiteSpeedのcPanelプラグインの権限割り当て不備で、CVSSは10.0。攻撃者はroot権限で任意スクリプトを実行できる。cPanelは5月19日の緊急パッチで該当プラグインを自動アンインストールする対応を取った。

#### 悪い例（背景・技術解説で肥大化）
> CVE-2026-48172は、LiteSpeed社が提供するcPanelエンドユーザー向けプラグインに存在する権限割り当ての不備で、CVSS最大値の10.0が付与された深刻度最大の脆弱性。攻撃者は本来許可されていない操作をroot権限で実行でき、サーバ全体の支配権取得に直結する。記事によると本脆弱性は実環境で能動的に悪用されており……（以下400字続く）

### コミュニティ反応（任意）
- `last30days-skill` で見つけた Reddit / Hacker News / GitHub / YouTube / X などの反応は、通常ニュース本文とは区別して任意で付ける
- 書き出しは `補足:` または `コミュニティ反応:` にする
- 反応は「議論が多い」「導入時の懸念が出ている」「利用者が比較している」のように、反応であることが分かる表現にする
- 反応補助ソースにも Markdown リンクを付ける
- 事実確認が必要な内容は、主要信頼ソースまたは公式情報のリンクを通常ニュース側に置く
- 反応補助ソースだけで構成される記事は、通常ニュースとしてではなく反応欄として扱う

### リンクの要件
- リンクは必ず Markdown 形式 `[ソース名](URL)` で記述する。素のURL（`https://...` のみ）は `<details>` ブロック内でクリックできないため使用禁止
- リンク先は**個別記事のURLのみ**とする。以下は使用禁止：
  - ニュースルーム/一覧/トップページ（例：`https://www.anthropic.com/news`）
  - タグ・カテゴリページ（例：`.../tag/ai/`）
  - 検索結果ページ
- リンク先ページで**公開日が確認できること**。公開日が不明な記事は採用しない
- 採用時点で **7日以内**（直近1週間）に公開された記事のみ採用する
- **各リンク行の末尾に、確認した公開日を `<!-- pub:YYYY-MM-DD -->` 形式の HTML コメントで必ず付与する**（例：`→ [TechCrunch](https://...) <!-- pub:2026-06-09 -->`）
  - このコメントは GitHub Pages のレンダリングでは表示されず、読者には見えない
  - `cowork/scripts/check.sh` がこのマーカーを基準日と突き合わせ、7日以内かをオフラインで機械検証する（マーカーが無いと鮮度を検証できず WARN になる）
  - 通常ニュースの主要リンクだけでなく、反応・補助ソースのリンク行にも付ける
- 反応補助ソースも個別URLを使う。Redditのサブレディット一覧、GitHubのリポジトリトップ、YouTubeチャンネル、Xプロフィールなどは原則として使わない

### 整合性チェック（保存前）
- まとめ本文と `<details>` の内容が一致していること
- `（N件）` の数字が `<details>` 内の実際のリスト項目数と一致していること
- 0件カテゴリの `<details>` ブロックが削除されていること
- すべてのリンクが個別記事URLであること
- **すべてのリンク行に `<!-- pub:YYYY-MM-DD -->` の公開日マーカーが付き、いずれも基準日から7日以内であること**
- 反応補助ソースが、通常ニュース本文の事実確認リンクとして単独利用されていないこと
