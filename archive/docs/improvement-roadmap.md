## B. 週次ファクトチェック（digest 内容の裏取り）

**背景**: 日次 `check.sh` は構造・鮮度・ソースを検証するが digest 内の数値・固有名詞の
真偽は見られない（2026-06-10 に資金額の誤り・CEO名の捏造が発生）。これを事後監査で補う。

**設計判断（2026-06-21、ユーザー決定）**:
- **頻度**: 週次（日次の公開を止めない／重くしないため独立ジョブ化）。日次予防ルール(B2)は入れない。
- **カバー範囲**: 高リスク抽出＝**厳しめ**（金額/%/CVE/CVSS）。実測 約25記事/週（全約71中）。月〜0.35〜0.45M tokens。
- **修正**: 明確な誤り（WRONG）かつ正しい値が出典で確定できる場合のみ、**該当箇所だけ**最小編集。
  曖昧(UNCERTAIN)は直さずレポート。

**成果物**:
- `cowork/scripts/extract_claims.sh` — 高リスク主張をオフライン抽出（厳しめフィルタ）。LLM 照合を
  抽出記事のみに限定しコストを抑える。フィルタ強度はスクリプト先頭の `RISK` で調整可。
- `cowork/WEEKLY_FACTCHECK.md` — 週次ジョブの手順（抽出→照合→対象のみ修正→`check.sh`再検証→
  レポート→commit）と Cowork scheduled task 用プロンプト。

**残作業（ユーザー側）**: Cowork で週次 scheduled task を登録（日次タスクと同様）。

**留意**: 公開→修正に最大7日の時間差が出る。固有名詞の捏造は抽出器では拾えないため、抽出された
記事は digest 全体（数値＋固有名詞）を LLM が照合する設計。

## 全体方針：土台 → 防御 → 拡張 → 体験

5軸はフラットではなく依存関係がある。ソース拡充も自動化の信頼も、`check.sh` と
SOURCES/DENYLIST が**単一の正**であることに依存する。土台がブレたまま積むと崩れるため順番が重要。

```
[Phase 1 土台]  構造検証 + 単一の正           ← 最大ROI・最優先（着手中）
      ↓
[Phase 2 防御]  運用フォールバックの明文化     ← 完了
      ↓
[Phase 3 拡張]  ソース拡充
      ↓
[Phase 4 体験]  UI（RSS → アーカイブ整理 → 検索）
```

UI の RSS は独立した quick win のため、いつ着手してもよい。

## 実装着手前の注意（恒久メモ）

- ローカルリポジトリは GitHub と乖離しやすい。実装前に必ず `git pull --ff-only` で最新化する。
- 日次運用を止めないこと。`check.sh` 変更時は既存 `_posts` 全件で回帰確認してから push する。
- 明日の出力で検証する変更は、前夜までに GitHub `main` へ push されている必要がある
  （日次生成は Cowork が GitHub から clone して走るため）。
