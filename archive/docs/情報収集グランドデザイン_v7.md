# 情報収集グランドデザイン v7

## 背景と課題

PMも兼ねるEM/プログラマーとして、以下の課題を抱えている。

- 情報ソースがAI・技術系ブログに偏っており、カバーすべき範囲が拾えていない
- 「拾えていない情報がある」ことへの不安が大きい
- 能動的な情報収集の仕組みがなく、Googleニュースのアルゴリズムに依存している
- 情報収集に構造（いつ・何を・どう）がない
- 読んだ情報が流れていき、ストックされない

---

## カバーすべき6領域

| # | 領域 | 現在の充足度 | 必要な深さ | 目的 |
|---|------|-------------|-----------|------|
| 1 | 一般ビジネスニュース | ✕ ほぼ追えていない | 見出しレベル | 社会人として会話についていける教養 |
| 2 | AI最新動向 | ◎ 追えているが網羅・ストックしたい | 深い | 技術選定・プロダクト判断の基盤 |
| 3 | 新サービス・ローンチ情報 | △ 追えていない | 中程度 | 市場感覚・競合理解 |
| 4 | 国内技術・ツール | △ レイヤー1で断片的に触れている | 中程度 | 国内の実践的な技術情報・ツール発見 |
| 5 | EM/PM系（マネジメント・プロダクト） | △ 追えていない | 中程度 | ロールに直結する知識 |
| 6 | セキュリティ | △ 追えていない | 重要なもののみ | リスク管理・実務上の判断 |

## レイヤー2：朝のブリーフィング（自動生成パイプライン）

### ニュース収集方針

**コアコンセプト：信頼できるソースを固定し、その更新を追う。**

AIによる重要度判定に頼るのではなく、ソースを厳選することで品質を担保する。全ソースの取得はCoworkのWeb検索で統一し、シンプルな構成とする。

## 技術構成

### Cowork スケジュールタスク

- **頻度：** 毎朝（平日 or 毎日、運用で調整）
- **スキル：** ブリーフィング生成スキル（SKILL.md）で品質を固定
- **処理内容：**
  1. 固定15ソースの最新記事をWeb検索で収集
  2. スキルに従って6カテゴリ別に整形・日本語要約
  3. Markdownファイルを日付付きで生成（例：`2026-03-28-briefing.md`）
  4. GitHubリポジトリに `git commit && git push`

### GitHub リポジトリ（パブリック）

```
daily-briefing/
├── _config.yml              # Jekyll設定
├── _layouts/
│   └── default.html         # ページテンプレート
├── _posts/                  # ブリーフィング記事（Jekyll規約）
│   ├── 2026-03-28-briefing.md
│   ├── 2026-03-27-briefing.md
│   └── ...
├── index.md                 # トップページ（最新記事一覧）
├── .github/
│   └── workflows/
│       └── deploy.yml       # GitHub Actions
└── README.md
```

### GitHub Actions

```yaml
on:
  push:
    branches: [main]

jobs:
  build-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build with Jekyll
        uses: actions/jekyll-build-pages@v1
      - name: Deploy to GitHub Pages
        uses: actions/deploy-pages@v4
```

### 静的サイトジェネレーター：Jekyll

GitHub Pagesネイティブサポート。設定最小限。日付ベースの記事管理がブリーフィングに最適。

### 閲覧

- **PC：** `https://<username>.github.io/daily-briefing/`
- **スマホ：** 同URLをブックマーク or ホーム画面に追加
- **過去記事：** 日付ごとに一覧表示、Git履歴で全件保持

## GitHub Actions コスト検証結果

**結論：完全無料で運用可能。**

パブリックリポジトリでの標準GitHub-hostedランナー使用は無料かつ無制限。今回の用途（1日1回のJekyllビルド、数十秒程度）では課金が発生する要素はゼロ。GitHub Pagesの配信も無料。

## この構成で解決される課題

| 元の課題 | 解決方法 |
|---------|---------|
| ビジネスニュースが拾えていない | 日経・Reuters等のソースを固定してカバー |
| 興味のある分野に集中しすぎる | 6カテゴリ × 15ソースを固定し、偏りをシステムで防ぐ |
| 何を追うべきか不明確 | 6領域を明示的に定義済み |
| 情報が流れていく（ストック問題） | GitHubリポジトリに全履歴が残る |
| 仕組みがなくアルゴリズム任せ | 固定ソース＋Coworkスケジュールで毎朝自動実行 |
| モバイルで見れない | GitHub Pagesでどこからでも閲覧可能 |
| AI動向がストックされない | ブリーフィングに含めることでレイヤー1と合わせて二重カバー |

## 制約と対策

| 制約 | 影響 | 対策 |
|------|------|------|
| Coworkの実行にはPC起動中 & Desktopアプリが必要 | 朝PCを開かないとブリーフィングが生成されない | OS起動時にClaude Desktopを自動起動に設定 |
| 有料記事はスクレイピングできない | 日経等の有料記事は見出ししか取れない可能性 | 見出し＋公開部分の要約で対応。深読みは手動で |
| ソースの更新頻度にばらつき | LeadDev等は毎日更新ではない | 「更新があれば」の条件付きで対応 |
| Web検索ベースのため取得結果にブレがある | 日によって拾える記事が変わる | 運用で許容。将来的にRSS事前フェッチで改善可 |
| Jekyll の `<details>` 内Markdownレンダリング | デフォルトでは正しく処理されない場合がある | `markdown="block"` 属性を使用、または `_config.yml` で CommonMarkGhPages を設定 |

## 将来の最適化

運用が安定したら以下を段階的に検討：

**トークン最適化：**
初期はWeb検索ベースで統一するが、運用データを見ながらより良いアプローチを模索する。cron事前フェッチは選択肢の一つであり、確定ではない。RSS対応10ソースを活用したトークン削減の可能性を検証しつつ、最適な方法を選定する。

**その他の拡張案：**
- 品質フィードバック：ブリーフィングにいいね/不要マークを付けて改善ループ
- 週次まとめ：日次ブリーフィングを週末に統合サマリー化
- カテゴリ検索：Jekyll のタグ機能で過去記事をカテゴリ別に閲覧
- ソース追加・入れ替え：運用で物足りないカテゴリがあればソースを追加
